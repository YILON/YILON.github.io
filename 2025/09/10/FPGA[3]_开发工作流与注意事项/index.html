ILA-在线抓取波形（Debug）

vivado在线抓取波形是通过ILA IP实现的。
虽然可以配置ILA IP，但是这样不仅麻烦，而且可能会使用错时钟，造成跨时钟域的不稳定。
因此，一般推荐让vivado自动配置。

方法一：手动添加信号列表

在综合完成后，点击综合流程中的Set up Debug，跟着向导选择想要抓取的信号即可。
它会自动识别信号所在
记得在工具配置完成后，Ctrl + S保存当前的配置。

ILA本质上就是一个IP，因此“抓取信号”这个行为会影响综合生成的电路逻辑，
比如一些信号本应被优化掉的，但因为被ILA使用而保留了。

该操作本质上是在用户约束文件.xdc中写入相关的命令，因此如果信号发生了修改或想删除ILA IP核时，直接删除.xdc中相关的命令即可。

约束

时序约束

时序约束的目标是让综合与布局布线工具了解设计者的时钟架构、信号关系及外部接口要求，从而在物理实现中满足时序要求，避免亚稳态和功能异常。

主要目标：

  • 定义时钟及其关系
  • 限制输入/输出接口的时序
  • 约束异步信号，避免误判
  • 指导工具进行优化

时钟约束（Clock Constraints）

  • create_clock：定义主时钟源的频率和波形。

对于差分时钟，只需要对_p进行创建即可。


  • create_generated_clock：定义由主时钟分频、倍频或移相后生成的派生时钟。

定义派生时钟的场景有：

  • 设计里有分频逻辑（fabric divider），例如用触发器做的 /2、/3 分频（这种做法并不推荐），或者用 BUFG_DIV / BUFGCE_DIV 等硬件分频器输出。
    如果没有声明，STA 可能把分频后驱动的寄存器当作“未约束/异步”寄存器，导致错误的 timing 报告或者漏掉需要的约束。

  • 时钟通过逻辑产生（例如由一个寄存器的 Q 输出驱动下一级时钟树） —— 需要把该输出声明为派生时钟的来源（-source 指向那个 pin/net）。

  • 时钟发生器（PLL/MMCM/ALTPLL 等）产生的输出，（有些时钟IP在生成时会把约束也做好）。

简言之：凡是有“一个时钟衍生出另一个时钟”的情况——并且衍生时钟驱动寄存器或关键路径——就应该考虑 create_generated_clock。


  • set_clock_groups：声明不相关时钟组，避免工具错误分析。

输入/输出延迟约束（I/O Delay Constraints）

  • set_input_delay：定义外部数据输入相对时钟的建立/保持时间。
  • set_output_delay：定义 FPGA 输出数据被外部器件采样的时序需求。

异步与多周期路径（Exceptions）

  • set_false_path：声明无需进行时序分析的路径（如异步 CDC）。
  • set_multicycle_path：声明某路径的有效时钟周期数大于 1。
  • set_max_delay / set_min_delay：对特殊路径设置特定延迟约束。

其他约束

  • case_analysis：固定输入信号的逻辑状态以优化时序。
  • max_fanout：限制驱动扇出以提高时序性能。

时序分析与收敛

  • 静态时序分析（STA）：工具报告建立时间（Setup）与保持时间（Hold）裕量（Slack）。
  • 关键路径分析：找出最长延迟路径，优化逻辑或布局。
  • 迭代优化：通过 RTL 优化、约束调整或工具参数优化实现时序收敛。

常见问题与解决方法

问题 可能原因 解决思路
未收敛 逻辑路径过长 插入流水线、优化 RTL、使用高速资源（DSP/BRAM）
Hold 违例 时钟树偏差或布局过近 添加延迟、约束优化
未约束路径 时钟或 I/O 未定义 补充时钟/I/O 约束
误用 false_path 真实路径被忽略 审查约束，严格控制例外

物理约束

对于差分信号，只需要对_P进行配置即可，_n会自动绑定。
注意同一个bank的IO电平要选择一致（和VCCIO的输入相同）。

布局布线

不同IDE的工作流可能不一样，这里只说明vivado在Implementation的工作流。

  • 设计初始化 (init _design)

  • 优化设计 (opt_design)
    • 逻辑优化：执行常量传播、移除冗余逻辑、重组逻辑以减少层级，目标是减少资源使用和提高时序性能。
    • LUT 融合：将多个小的 LUT 合并成一个更大的 LUT，以节省资源。
    • 寄存器优化：复制高扇出的寄存器（以减轻布线负担）、移除多余的寄存器、根据约束调整寄存器的控制集（时钟使能、复位类型）。
    • 时序驱动优化：根据时序约束，对关键路径进行初步优化。

  • 功率优化 (power_opt_design)
    • 对于功耗不敏感的设计可以跳过。

  • 布局 (place_design)
    • 全局布局：首先进行粗略的、快速的布局，将所有单元放到芯片的大致区域，目标是最小化所有连接线的总长度。
    • 详细布局：在全局布局的基础上，进行精细调整，将单元精确地放置到特定的 Slice 中。这个过程严格遵守 FPGA 的布局规则。
    • 时序驱动布局：布局器会优先考虑时序约束。它会努力将关键路径上的逻辑单元彼此放置得更近，以减少信号延迟。
    • 拥塞避免：布局器会尽量避免将过多逻辑挤在一个小区域内，否则会导致后续布线困难。

  • 布局后功率优化 (power_opt_design)
    • 在布局之后，基于更精确的线负载模型，进一步进行功耗优化。

  • 布局后物理层优化 (phys_opt_design)
    对 setup time 违例的关键路径进行优化，对于实现时序收敛至关重要，可以显著减轻后续 route_design 的压力.例如：
    • 逻辑重组：改变逻辑结构以减少 LUT 的级数。
    • 寄存器复制：复制高扇出的驱动寄存器，减少每个寄存器的负载，改善延迟。
    • 位置微调：在局部范围内移动单元的位置以改善时序。
    • 缓解拥塞：对高拥塞区域的逻辑进行扩散或优化，以降低布线难度。

  • 布线 (route_design)

  • 物理层优化 (phys_opt_design)
    在布线完成后，基于最终、最精确的时序信息（包含实际布线延迟），
    对无法闭合的时序进行最后的修复。此时看到的时序违例是必须解决的。

生成bit流

默认不配置的情况下，
vivado的Generate Bitstream生成的是只能烧录进FPGA RAM（掉电失效）中的.bit文件。

该.bit文件放在‘工程名’.runs/impl_1中。


能固化到flash的bit流有两种：.bin和.mcs。

.bin文件是未经处理的原始二进制数据。它不包含地址信息或其他元数据，只是按顺序排列的字节流。
因此在烧录.bin文件时，需要手动指定烧录起始地址。


.mcs文件是 ASCII 文本文件，它将二进制数据编码为十六进制字符。
该文件内部包含了地址信息。每一行都指定了数据应该被写入存储器的哪个地址，这使得它可以处理非连续的数据块。
这种格式使得烧录过程更加健壮，因为烧录工具可以根据文件中的地址信息将数据准确地放置到存储器的正确位置。


生成并使用bin文件

点击上方工具栏的Tools -> Settings -> Bitstream，找到并勾选-bin_file*，
这样vivado在Generate Bitstream时会生成对应的bin文件。

当bit流生成完成后，连接上Jtag，点击open hardware manager -> Auto connect，
正常情况下会识别到FPGA器件。
右键该器件，选择Add Configuration Memory Device，选择实际对应的flash芯片即可。
界面中该FPGA器件下面会出现对应的flash器件，右键该flash器件，
选择Program Configuration Memory Device，再选择对应的.bin文件，烧录即可。

如果不选择指定的flash器件进行program，那么默认情况下只会烧录到FPGA RAM中。

生成并使用MCS文件

在生成.bit文件后，选择上方工具栏的Tools -> Generate memory Configuration file，
勾选Memory Part，根据实际使用的flash器件配置该行；
在Filename中输入生成文件的路径和文件名；
勾选Load bitstream files，下方选择对应生成的.bit文件，
点击ok即在指定路径中生成.mcs文件。

同烧写.bin文件一样的步骤即可将该.mcs文件烧写进外部flash中。

以上步骤实际上是将.bit文件转换为.mcs文件，和Generate Bitstream并不相关，
因此每次生成新的.bit文件都需要如此操作生成新的.mcs文件。


TCL脚本

在vivado中可通过TCL指令或脚本.tcl执行一些操作。
有时候希望在执行某个步骤（比如布线前）前执行某个脚本.
这可以在Project manager -> setting中配置。

FPGA上电处理

FPGA在上电时会执行以下逻辑：

  1. 供电上升
    主要电压：核心（VCCINT / VINT）、辅助（VCCAUX）、IO（VCCIO/Bank rails）等。
    要点：必须按厂家 datasheet 的允许范围 & 建议顺序和允许斜率上升。若电源时序错误可能导致不可预测行为。
    输出：Power-good（PG）标志或电源管理 IC 的信号通常用于后续 POR 判断。

  1. Power-On Reset（POR）
    POR 由电源管理或 FPGA 内部产生，用来在电源不稳时把逻辑保持在复位状态。
    POR 保持期间：配置逻辑通常不能开始。POR 会把某些内部状态保持/初始化。

  1. 检查 PROGRAM_B / INIT_B
    PROGRAM_B（通常为低有效）被外部拉低，可强制进入重配置；释放后 FPGA 重新走配置流程。
    INIT_B：配置子系统用来指示配置初始化是否完成；低表示未就绪或内部错误。

  1. 配置（bitstream 加载）
    接口：JTAG、SPI（master/slave）、SelectMAP、BPI 等，具体由器件与设计决定。
    期间：FPGA 内部配置引擎读取 bitstream，下载到配置内存（或直接编程在 eFUSE /配置 FLASH）。
    信号：CCLK（配置时钟）、DATA（配置数据），INIT_B 可能在此期间短脉冲，DONE 最终上拉。

  1. 配置内部自检 & 初始值写入
    配置引擎完成 CRC、初始化 LUT/BRAM 初值（来自 bitstream）、PLLs 初始化等。
    注意：BRAM/LUTRAM 的初始内容通常是从 bitstream 加载，而不是由 GSR 直接清零。

  1. GSR / GTS 的作用与释放
    在配置过程中，FPGA 会 断言 GSR（Global Set/Reset）来保证 flip-flop 进入已知初值（由 bitstream 中配置的 INIT 值决定）。同时 GTS（Global Tri-State）会 tri-state 大部分 IO，防止噪声。
    当配置成功（DONE 上升且 INIT_B/其他自检条件满足）时，配置逻辑会 释放 GSR/GTS，从而允许寄存器走向可用状态与 IO 恢复驱动能力。

  1. 用户复位管理（Reset Manager）
    尽管 GSR 已释放，仍建议用一个 Reset Manager：它会等待 power_good && done && pll_locked && other_peripherals_ready，然后逐时钟域同步地释放用户复位。
    要点：每个时钟域都要用复位同步器（2-stage 或 3-stage），异步断言、同步释放 是常见且安全的模式。

  1. 进入应用逻辑
    在各时钟域复位同步完成并确认外设/通信链路稳定后，应用逻辑开始运行。

FPGA的复位逻辑的处理

据赛灵思此处而言，
时序逻辑推荐使用同步复位。

应注意的是，FPGA上电的GSR并不是常规意义上的“复位”，这只是给每个寄存器在上电时有一个确定的初始值。

通常FPGA的rst_n信号是通过外部的IO输入-机械按钮输入的，想要在FPGA上电时自动执行一遍期望的复位逻辑，需要编写这样的逻辑。
