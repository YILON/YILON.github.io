初始化Flash并将函数放到RAM中运行

参考链接

上电复位时C2000 DSP的FLASH等待状态是很长的，需要进行配置以提高取指性能。
而这都是通过调用InitFlash()实现的，而该函数必须要加载进RAM中运行。

不同的DSP有不同的调用方法，比如F28377D已经封装好相关内容，只需要在project属性中预定义_FLASH宏即可。

但即使优化了FLASH的等待状态数，在FLASH中取指的速度仍然是低于在RAM中取指的，
因此对于速度要求特别高的函数，可以将其加载到RAM中。

由于以前的文章已经说过如何进行配置，这里不再重复。




浮点运算加速

C2000 DSP RTS库

在CCS中打开一个project，
右键project->properties->General->Runtime support library可以选择所需的支持库，
这些库与编译器相关。

  • rts2800.lib：基本运行时支持库。

  • rts2800_ml.lib：中等优化、Large Memory Model支持库（仅对整数和定点运算）

    开优化等级最好使用volatile限定变量，避免出现程序不按预期情况执行。
    也可以使用宏指令指定某段函数不使用优化：

  • rts2800_eh.lib：支持异常处理的基本运行时库。

  • rts2800_fpu32.lib：支持FPU32位浮点运算的运行时库。
    （在project的属性中配置–float_support=fpu32时自动链接）

  • rts2800_fpu64.lib：支持FPU64位浮点运算的运行时库。
    （在project的属性中配置–float_support=fpu64时自动链接）

FPU32和FPU64是不同的浮点单元，比如F2833x和F2837x的浮点单元都是FPU32，
而F2838x的浮点单元是FPU64。

小内存模型和大内存模型

Large Memory Model 是指 C28x 编译器的内存模型，
它决定了如何访问数据和程序代码，特别是如何处理大地址空间和数据指针。

在 C28x 平台上，
内存模型主要分为 Small Memory Model 和 Large Memory Model 两种，
区别在于如何使用指针和编址空间：

  1. Small Memory Model
    • 特点：
      假定所有数据和程序代码都能被 16 位地址表示。
      数据指针和函数指针均为 16 位。
      适用于内存需求较小的应用程序。

    • 限制：
      最大支持的地址空间为 64 KB（2^16）。
      对于代码和数据存储要求较大的项目可能不够用。

    • 优势：
      数据访问和函数调用效率高，因为使用的是 16 位指针。

  1. Large Memory Model

    • 特点：
      支持访问超过 64 KB 的地址空间。
      数据指针和函数指针扩展为 32 位。
      可以处理更大的数据集和更复杂的程序。

    • 实现细节：
      使用 32 位地址（上 16 位是页地址，下 16 位是偏移地址）访问内存。
      函数指针和数据指针占用更多的空间，
      但能够访问全范围的内存。

如果选择使用 rts2800_ml.lib 并启用 Large Memory Model，需要做以下配置：

  1. 启用 Large Memory Model
    在 CCS 中，可以通过以下步骤启用 Large Memory Model：
    打开 Project Properties -> Build -> C2000 Compiler -> Processor Options。
    找到 Memory Model 选项，并选择 Large Memory Model。

  1. 指针大小变化
    使用 Large Memory Model 时，指针大小会变为 32 位。这会带来以下影响：
  • 内存占用增加：指针占用的存储空间比 Small Memory Model 增大。
  • 访问效率稍低：因为需要处理 32 位地址，数据访问可能略慢。

  1. 链接器文件调整
    确保链接器文件（.cmd 文件）能够正确定义和分配大容量的内存块。
    如果程序代码和数据需要跨越多个内存段，
    需在链接器文件中明确配置这些段的地址范围。

使用FastRTS库加速浮点运算

参考链接

上面提到，对于拥有FPU的C2000 DSP，
TI提供了rts2800_fpuxx.lib用以支持 FPUxx 的浮点运算。

为了提高部分浮点三角函数运算速度，在controlSUITE还提供了 rts2800_fpuxx_fast_supplement.lib。
该库重实现了C的相关数学函数，在牺牲一定精度的情况下提升计算速度。

库位置放在C:\ti\controlSUITE\libs\math\FPUfastRTS中。
将此库添加进project后，记得配置库索引路径。

注意，在链接时此库必须先于rts2800_fpuxx.lib，
这需要在右键project->properties->Build->Link Order中进行指定，
rts2800_fpuxx_fast_supplement.lib需要排在rts2800_fpuxx.lib之前。

导入库文件与头文件fastrts.h之后，引入fastrts.h与math.h即可使用。

C28x_FPU_FastRTS.h是旧的头文件，被fastrts.h取代了，不需要用。

TMU浮点运算加速

参考链接

TMU即三角数学单元，（但应用不限于三角函数）
能实现比rts2800_fpu32_fast_supplement.lib快非常多的数学函数计算速度。
（个位数的流水线周期，但拥有同样的精度损失）

因此对于拥有TMU的DSP，没必要再使用这个库。
注意不是所有DSP都有TMU（比如F2833x），即使在project中可设置，但实际是无效的。
F28x7x拥有TMU。


实际上有两类TMU：TMU0和tmu1.
tmu1只适用于EABI格式；
tmu1支持tmu0的所有内容，还额外支持LOG2F32与IEXP2F32运算。
F2837xD只有tmu0。

TMU的支持说明在C2000编译器说明中

要启用TMU，在右键project->properties->Build->Processor Options中，
对Specify TMU support选择tmu0或tmu0即可，其会自动启用fpu32。

fp_mode

在右键project->properties->Build->Optimization中，
有一项是关于fp_mode的设置：

  • 严格(strict)(默认)

  • 宽松(relaxed)
    会使双精度浮点计算和存储在可能的情况下转换为单精度浮点。
    这种行为不符合 ISO 要求，但会加快代码速度，准确性会有降低。
    具体的：

    • 除以一个常数被转换为逆乘法
    • 如果可以，一些 C 标准浮点函数
      如 sqrt、sin、cos、atan、atan2 和 fmodf会被重定向到优化的函数
    • 如果启用TMU，则将以下浮点运算替换为TMU版本：
      浮点除法、sqrt、sin、cos、atan 和 atan2。
    • 如果使用tmu1，则使用以下 32 位 RTS 数学函数的宽松版本：
      exp2f()、expf()、log2f()、logf() 和 powf()
      注意，未提供适用于 double 类型的宽松版本。

内联函数

内联函数扩展可以通过消除函数调用开销来加快执行速度。
这对于经常被调用的非常小的函数特别有用。
函数内联涉及到在执行速度和代码大小之间进行权衡，
因为代码在每个函数调用点都是重复的。
在许多位置被调用的大型函数不适合内联。

过多内联会使编译显著变慢并降低所生成代码的性能。

具体影响见此文的2.11节。

注意事项

volatile关键字的使用

main()与中断或中断与中断之间共享的全局变量都应声明为 volatile
在访问表示内存映射外设的内存位置时，必须使用 volatile 关键字。

防止中断打断读/写

原子访问不会被中断打断。以下操作是原子的：

  • 16 位读/写
  • 32 位浮点读/写
    例外：写入 32 位浮点常量只有在使用单个操作码执行时才是原子的。
  • 使用单个操作码的32 位整数读/写

对于大于32位的操作，应该先禁止中断，避免操作被打断，
导致操作不完整。

C28x默认不启用中断嵌套，
因此在中断中执行的非原子操作不用担心被打断。

其它

取自此处

  1. 调用全局变量比调用局部变量快。

  2. 用指针进行顺序读写比用数组下标更快

  3. 对于多重循环，应将长循环置于内层，短循环置于外层。
